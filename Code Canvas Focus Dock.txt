Motivation
When developers open a new codebase, they need to quickly understand its structure without manual exploration. This system solves that problem by automatically scanning project files and building an interactive architecture map. Instead of manually hunting through directories, users get an instant visual representation of their code organization, import relationships, and key symbols.

Details
The workflow begins when a user selects a project directory through the native file dialog [2a]. The system then recursively walks the project tree, filtering for relevant file extensions like .js, .ts, .py, and .go while ignoring common directories like node_modules and .git [2d]. For each valid file, it reads the first 5KB of content and extracts function/class definitions using regex patterns to build a symbol index [2e].

The core innovation is the structural codemap generation that happens without any AI API calls [2c]. Files are automatically grouped into logical areas based on their directory structure, and import relationships are parsed to create connection edges between files. This creates an immediate, free architectural overview that works offline.

The entire process is security-hardened with path sandboxing to prevent directory traversal attacks, and results are cached to avoid re-scanning on subsequent opens. The final output is a JSON codemap with positioned nodes and edges that the frontend canvas can render interactively.

Details

The Focus Dock switches to code mode [3d] and calls openFile() which makes a secure IPC request to read the file content [3e]. The preload bridge enforces security by only allowing reads within the project boundaries. Once the content loads, the editor automatically scrolls to the exact line number and highlights the symbol range, creating a seamless transition from visual exploration to code inspection.

This flow demonstrates the coordinated panel architecture where canvas selection, project tree highlighting, and code editor loading all happen in perfect sync to maintain developer context.

Motivation
The Code Canvas Focus Dock needs a real-time AI chat system that can provide contextual assistance while users explore code architecture. The challenge is creating a responsive chat experience that streams responses instantly, maintains conversation context, and integrates seamlessly with the code visualization workflow. Details
The chat flow begins when a user sends a message through the Focus Dock interface [4a]. The request travels to the main Electron process via IPC, which maintains security by keeping AI API calls off the renderer. The main process delegates to the AI service, which establishes a streaming connection to OpenAI-compatible providers [4c].

The streaming architecture is key to user experience - instead of waiting for a complete response, the system processes chunks as they arrive [4d] and immediately forwards them to the frontend [4e]. This creates the illusion of instantaneous response while the AI is still generating content. The frontend renders each chunk in real-time, allowing users to see the AI "thinking" and providing early value before the full response is ready.

The system carefully manages conversation history and project context, attaching selected file contents and codemap data to each request so the AI can provide relevant, project-aware assistance. All of this happens through a secure IPC bridge that protects API keys and file system access while maintaining the responsive feel users expect from modern chat interfaces.

Motivation
The application needs to store sensitive AI API keys securely while maintaining user convenience. Storing API keys in plain text is a security risk, but users also need to configure different AI providers and models. The settings system solves this by providing encrypted storage with OS-level protection when available, falling back to file permissions when necessary.

Details
Settings management flows from the frontend form [6a] through a secure encryption pipeline. When you save settings, the main process validates the API key input [6b] and checks if your OS provides hardware-level encryption [6c]. If available, it uses Electron's safeStorage to encrypt the key [6d]; otherwise, it falls back to base64 encoding. The encrypted data is then written to a settings file with restrictive 0o600 permissions [6e], ensuring only the application can read it. The settings path is stored in your user data directory [6a], keeping configuration separate from project files while maintaining security boundaries.
